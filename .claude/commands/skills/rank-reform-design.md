# 改修設計スキル（rank-reform-design）

ステータス：開発中

---

## 目的
競合分析・自社記事分析の結果を踏まえて、網羅性改修後のH2/H3構成と各ブロックの方針を確定する。
このスキルの出力が、社員によるHTMLエディット（rank-edit-html）へのインプットになる。

## 使用するフロー
- 順位施策（網羅性改修フェーズ）
- インターンは このスキルまでを担当。出力は引き継ぎシートに含める

---

## インプット
以下の情報を確認してから作業を開始する。
前のスキルや依頼書から提供されている場合はそれを使う。
ない場合はユーザーに確認する。

- メインKW・サブKW・検索意図
- 自社記事のH2/H3構造・トピックインベントリ・構造上の問題
- 競合分析の差分整理（自社にない・粒度負け・自社のみ・同程度）
- 網羅性改修の規模（大：構成から変える / 小：部分的な追加）※rank-sc-checkの診断があれば使う

**ブロック別ライティングルール（必須）：**
設計前に `knowledge/block-rules.md` を読む。STEP3の確認ポイント・STEP5のSo what確認はblock-rulesと照合する。

**記事タイプの前提ルール（必須）：**
対象記事が地域記事の場合、`knowledge/article-types/local.md` を読んでから設計を始める。
構成案がルールに違反していないかを出力前に確認する。

---

## 実行フロー

### STEP1｜差分の採用判断
競合分析の差分整理を受け取り、各トピックを以下の4分類に振り分ける。

| 分類 | 定義 |
|------|------|
| 追加 | 自社になく、かつ採用根拠があるもの |
| 修正 | 自社にあるが検索意図との適合を高められるもの |
| 削除 | 狙いKWに対してノイズになるもの |
| 維持 | そのままで問題なく、自社の強みになるもの |

**採用基準：以下のいずれかが根拠として明示できるものだけを採用する。**
- SEO効果（検索意図への適合・KWカバレッジの改善）
- コンテンツ品質向上（ユーザーの理解・比較・意思決定に寄与する）

「競合がやっているから」は採用理由にならない。

### STEP2｜規模に応じた設計方針の確認
rank-sc-checkで診断した網羅性改修の規模を確認し、設計の方針を決める。

**規模：大（構成から変える）の場合**
- H2の追加・統合・削除・順番変更も含めて全体構成を再設計する
- GSCでBパターン（不要クエリ）が出ていた場合は、対応ブロックの削除・縮小を最初に処理する

**規模：小（部分的な追加）の場合**
- 既存のH2/H3構造を基本的に維持する
- 「追加」「修正」対象のブロックのみを変更する

### STEP3｜H2/H3構成の設計
STEP1・2の判断を踏まえて、改修後のH2/H3全体構成を出力する。

確認ポイント：
- メインKWが自然な日本語でタイトル・H1に含まれているか
- 各H2が検索意図の主要な問いに1つずつ対応しているか
- H3はH2の問いを掘り下げているか（H3で新たな問いを立てない）
- 重複・役割不明の見出しがないか
- まとめH2が記事全体の結論になっているか

### STEP4｜各ブロックの方針明示
H2/H3それぞれに以下を付記する。

- 分類ラベル：追加 / 修正 / 削除 / 維持
- 方針：何をどう変えるか（1〜2文）
- 根拠：なぜその判断をするか（検索意図・競合差分・品質観点のいずれか）

### STEP5｜So whatの確認
各H3に「ユーザーへの結論・判断支援」が含まれているかを確認する。

- 箇条書きや事実の羅列だけで終わっているH3はないか
- 「つまりユーザーはどう判断すればいいか」が書かれているか

---

## ユーザー確認
STEP3〜5の構成案をユーザーに提示し、確認を取る。

---

## アウトプット

```
【改修後タイトル案】

【H2/H3構成】
  H2：〇〇　[修正] 根拠：〇〇
    H3：〇〇　[追加] 方針：〇〇 / 根拠：〇〇
    H3：〇〇　[維持]
    H3：〇〇　[削除] 根拠：〇〇
  H2：〇〇　[維持]
    ...

【So what確認】
  要注意H3：〇〇（結論・判断支援が不足）
```

---

## 鉄則
- **順位施策の目的はメインKW・サブKWの順位改善。** 採用根拠は「このH2/H3を追加・修正するとメインKW・サブKWの順位が上がるか」で判断する
- 採用根拠を明示できないものは提案しない
- 「競合がやっているから」は理由にならない
- So whatのない箇条書きだけのH3は設計に含めない
- 構成確定前に本文を書き始めない。必ずユーザー確認を取る
